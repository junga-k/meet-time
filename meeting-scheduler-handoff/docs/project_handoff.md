# 프로젝트 현황 — 회의 일정 조율 앱 (와이어프레임 완성 · 구현 준비 완료)

> 이 문서는 지금까지의 설계 대화 전체를 이어서 작업할 수 있도록 요약한 현황 문서입니다. Claude Code에 구현을 맡길 때 이 파일을 가장 먼저 읽으면 됩니다. 17개 화면 와이어프레임 제작과 전체 통합 재검토·수정이 모두 끝난 최종 상태를 반영합니다.

## 프로젝트 개요

같은 회사 동료 6명 내외가 필수/선택 참석자와 온/오프라인 조건을 고려해 회의 시간을 빠르게 조율하는 앱. 핵심 차별점은 "여러 사람의 조건을 한 번에 이해하고 모두에게 괜찮은 시간을 자동으로 찾아주는 것" — 기존 캘린더 앱·단순 투표 도구와의 차이.

**플랫폼**: 반응형 웹 앱 (모바일 우선, 390×844px 기준. 상단 24px 상태바 + 하단 **56px** 탭바 + 좌우 16px 여백 공통 규칙 — 탭바 높이는 56px로 최종 통일)
**아이콘**: Tabler Icons 라인 스타일, 이모지 미사용. 사람 식별은 원형 아바타(사진 또는 이니셜)

## 핵심 설계 결정 요약

### 데이터 모델 (12개 엔티티)
`USER, MEETING, PARTICIPANT, TIME_SLOT, SLOT_RESPONSE, AGENDA, NOTIFICATION, ROOM, ROOM_BOOKING, MITIGATION_REQUEST, MEETING_NOTE, ACTION_ITEM`

상세 필드는 `db_model_mvp.md` 참고. 주요 필드만 요약:
- `MEETING`: mode(자동계산, nullable), stage(필수응답중/선택확인중/확정), duration_minutes(**주최자 직접 입력, 가변— 고정 1시간 아님**), candidate_start_date/end_date(캘린더 선택), required_response_deadline/optional_response_deadline(자동 역산), reschedule_count(자동1+수동2, 총 3회 한도 실제 강제)
- `PARTICIPANT`: role(필수/선택/**주최자**), confirmation_status(미확인/확정/불참 — '확정'은 시간 최종 확정 시점에 시스템이 자동 기록), attendance_mode(대면/온라인/무관, **슬롯 응답 다음 순서로 선택**, 마감 전까지 재수정 가능·재수정 시 모드 재계산), responded_at, decline_reason
- `TIME_SLOT`: is_shortlisted(상위 5개 압축 표시), end_time = start_time + `MEETING.duration_minutes`
- `MITIGATION_REQUEST`(신규): 완화요청(변경요청) 큐/이력. 사람 기준 순차 알고리즘으로 화면 6·7 통일
- `MEETING_NOTE`/`ACTION_ITEM`(v2→v1 승격): 화면 10(회의록·향후 추진 과제)의 데이터. 작성자 1인 전용 편집권한(주최자 기본, 재지정 가능) + 임시저장/등록 2단계

### 핵심 플로우 — 단계적 응답수집

```
로그인(13) → [최초] 프로필설정(14) → 온보딩(15) → 회의 목록(1)
회의 목록(1) → 회의 생성(2, 캘린더로 후보 기간만 지정)
  → 필수 참석자: 슬롯(시간대) 그리드 응답(3) → 참석형태 선택(대면/온라인/무관) → "응답완료"
  → (전원 응답완료 또는 마감 중 이른 시점) 상위 5개 후보 압축 + 회의 모드 자동계산
  → 선택 참석자: 압축 후보 확인(12) → 참석형태 선택 → 가능/불가 확인 → "응답완료"
  → (전원 응답완료 또는 마감 중 이른 시점) 확정
  → 모드가 오프라인/하이브리드면 회의실 자동 매칭+예약, 온라인 요소 있으면 링크 자동생성
  → 주최자가 대시보드(5)에서 승인/변경
  → 확정 후 참석자는 참석 재확인(8) → 회의 상세(9)
  → 회의가 시작된 시점부터 회의록·향후 추진 과제(10) 실시간 작성 가능(화면 9에서 진입)
```

**왜 이렇게 됐는지 (판단 근거, 발췌)**
- 필수/선택 참석자를 동시에 받으면 선택 참석자까지 전체 그리드에 응답해야 해서 피로도가 높고 시간도 오래 걸림 → 필수 먼저, 선택은 압축된 후보만
- 압축 후보 개수는 **5개**로 확정
- 마감은 상한선일 뿐, 대상자 전원이 "응답완료" 누르면 조기 진행
- 온/오프라인/하이브리드는 주최자가 미리 안 정하고, 참석자들이 고르는 참석형태(attendance_mode)를 집계해 자동 계산. 기준은 "온라인을 고른 사람이 있는지"가 우선 — 상세 규칙은 `db_model_mvp.md` 참고
- **회의 길이는 주최자가 직접 입력하는 가변값**(최대 8시간, 5분 단위) — 과거 "기본 1시간 고정"이었던 방침을 통합 재검토에서 폐기하고 가변 길이를 정식 지원하기로 확정. `TIME_SLOT`은 이제 `duration_minutes` 기준으로 구간을 계산
- 회의실은 후보 평가 때는 조회만, 확정 시점에만 자동 매칭+예약
- 선택 참석자가 압축 후보 5개 전부 "불가"면 `불참`으로 기록하고 진행. 이후 회의 시작부터 회의록을 확인할 수 있다는 안내 표시
- 확정 후 필수 참석자 불참 시: **자동 1회 + 수동 2회, 총 3회 한도**를 실제로 강제. 한도 도달 시 담당자 에스컬레이션 안내로 전환
- 완화요청(변경요청) 순차 알고리즘은 **사람 기준**(한 슬롯에 대해 후보자 순서대로 요청)으로 통일. 이력은 `MITIGATION_REQUEST` 엔티티로 저장
- 참석 재확인(화면 8) 불참 통보 시 사유 입력 필수. 필수 참석자면 "재조율 시작" 안내, 선택 참석자면 "회의 시작부터 회의록을 확인할 수 있어요" 안내(회의록 기능이 v1로 승격되어 실제 이동 가능)
- 회의록(화면 10)은 **회의 시작 시점부터** 실시간 작성 가능(회의 종료 후로 제한하지 않음). 작성자는 기본 주최자, 주최자가 재지정 가능. 편집권은 지정된 작성자 1인뿐(향후 추진 과제는 참석자 누구나). 저장은 임시저장/등록 2단계, 등록 시에만 참석자 전원에게 알림 발송

### 화면/UI 공통 규칙
- 모바일 세이프에어리어: 390×844px 고정, 상태바 24px + 탭바 **56px** + 좌우 16px
- 주요 액션(회의생성·응답완료·안건등록/수정/삭제 등 **데이터를 실제로 바꾸는 액션**)은 클릭 시 확인 팝업 한 번 더 (슬롯 토글, 데이터 변경 없는 조회성 클릭처럼 가벼운 조작은 제외 — 예: 완화요청에서 "불가로 유지"는 데이터 변경이 없으므로 확인 팝업 없이 즉시 처리)
- 참석자 검색: 이름 검색 드롭다운 + 부서 조직도 모달
- 참석자 추가 시 역할(필수/선택) 명시적 선택 필수 (기본값 없음)
- 안건 첨부파일: 첨부 시 파일명+삭제버튼이 위, "파일 선택" 버튼이 아래
- 알림 배지: 숫자 아닌 빨간 점만
- 토스트: 공용 `.toast`/`.toast.show` CSS + `showToast(msg)` JS 함수 패턴으로 전체 화면 통일(탭바+하단 CTA가 모두 있는 화면은 `bottom: 101px`, 그 외 사전 로그인 화면 등은 화면별 관례값 사용, `z-index: 45` 통일)
- 하단 CTA 버튼 영역(`.footer`): 스크롤 없는 화면은 구분선 없음, 스크롤 있는 화면은 "스크롤 중일 때만 동적 그림자"(`show-boundary` 클래스, 스크롤 이벤트로 토글) 방식으로 전체 통일
- 배지/경고 색상 팔레트: 제안중=파랑, 확정=초록, 재조율중=빨강, **재확인 대기=amber(신규 분리)**, 취소=회색. "주의/경고" 박스는 전체 화면에서 amber(`--accent-amber`/`--accent-amber-bg`/`--accent-amber-border`) 통일(화면 5가 빨강을 쓰던 것을 amber로 수정 완료)
- 사용자 노출 문구: "완화요청"(내부/시스템 용어) → 화면 표시는 "변경요청", "거절"→"불가"로 순화 통일(전체 화면 반영 완료)
- 날짜 표기: 회의 인스턴스 날짜는 `M/D(요일)`(예: "7/14(화)"), 절대 날짜를 다루는 예약/회의록류 화면은 `YYYY.MM.DD (요일)`로 통일

### AI 활용 범위
회의실 예약·모드계산·알림·리마인드는 전부 규칙 기반 자동화 (LLM 불필요, 예측가능성이 더 중요). AI(LLM)가 실제로 유용한 곳은 **회의록(화면 10) "AI 초안 생성"** — 작성자가 입력해둔 러프한 메모를 LLM(예: Claude API)으로 다듬어진 문단으로 재정리하는 용도(녹음/STT 기반 아님). 이 기능은 v2가 아니라 v1(MVP) 범위에 포함됨.

### 구현 시 특히 유의할 구조적 사항
1. **TIME_SLOT 길이가 가변**이 됨 — `MEETING.duration_minutes` 기준으로 `end_time`을 계산해야 하며, 슬롯 겹침/예약 로직이 "고정 1시간" 전제로 하드코딩되지 않도록 주의
2. **재조율 횟수 캡(3회)**과 **완화요청 사람 기준 순차 알고리즘**은 화면 5·6·7에 걸쳐 하나의 일관된 백엔드 로직/엔티티(`MITIGATION_REQUEST`)로 구현되어야 함(와이어프레임에서는 각 화면이 독립적으로 데모하고 있으나 실제로는 상태를 공유하는 하나의 프로세스)
3. **최종 확정 슬롯 선택**은 하드 제약 통과 후보 중 `unavailCount` 최소 기준 하나로 통일(화면 5의 자동 계산과 "강제 진행" 버튼이 같은 알고리즘을 써야 함)
4. **알림 라우팅은 수신자 역할에 따라 분기**해야 하는 케이스가 있음(응답요청/마감임박 → 필수 참석자=화면3, 선택 참석자=화면12, 주최자=화면5) — 알림 한 건이 받는 사람에 따라 다른 화면으로 연결될 수 있다는 점을 서버/라우팅 설계에 반영
5. **회의록(화면 10) 편집 권한**은 `MEETING_NOTE.author_user_id` 1인 전용 — 서버에서도 이 검증이 필요(다른 참석자가 API로 직접 수정 요청해도 거부해야 함), 단 향후 추진 과제(`ACTION_ITEM`)는 참석자 누구나 가능
6. **화면 1의 "회의록" 배지**는 `MEETING_NOTE.status`='등록' 여부로 판단하며, 이 배지가 있는 회의는 다른 라우팅 규칙보다 우선해 항상 화면 9로 이동

## Claude Code에 전달하는 방법
