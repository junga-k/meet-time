# 회의 일정 조율 앱 — 화면별 상세 스펙

> 데이터 모델(USER, MEETING, PARTICIPANT, TIME_SLOT, SLOT_RESPONSE, AGENDA, NOTIFICATION, ROOM, ROOM_BOOKING, MEETING_NOTE, ACTION_ITEM)과 핵심 사용자 플로우를 기준으로 작성. 와이어프레임 설계 시 참고 문서.

## MVP 범위

**플랫폼**: 반응형 웹 앱 (단일 코드베이스, 화면 크기에 따라 레이아웃 반응 — 모바일/태블릿/데스크톱 별도 앱 아님). 지금까지의 와이어프레임은 모바일 기준 레이아웃이며, 데스크톱/태블릿 레이아웃은 화면별로 이후 별도 검토

**아이콘 스타일 (모든 화면 공통)**: 라인 아이콘 (Tabler Icons, 이모지 미사용). 사람 식별은 별도로 원형 아바타 사용 — `USER.profile_image_url` 있으면 사진, 없으면 이름 첫 글자 이니셜 (추후 Slack·사내 메신저 연동 시 그쪽 프로필 사진으로 자동 동기화 검토 — v2)

**주요 액션 확인 팝업 (모든 화면 공통)**: 되돌리기 어렵거나 다른 참석자에게 영향을 주는 액션은 클릭 즉시 실행하지 않고, 요약 정보와 함께 "확인/취소" 팝업을 한 번 더 띄운다. 대상: 회의 생성("회의 만들기"), 슬롯 응답 완료("응답 완료" — 화면 3·12), 안건 등록/수정/삭제. 슬롯 셀 토글·참석자 역할 선택처럼 빠르고 되돌리기 쉬운 조작에는 적용하지 않음 (불참 통보는 이미 사유 입력 단계가 확인 절차 역할을 하므로 별도 팝업 생략)

**모바일 세이프에어리어 (모든 화면 공통 적용)**
- 기준 화면 크기: 390×844px (표준 아이폰 기준, 고정 높이 — 콘텐츠가 넘치면 카드가 늘어나는 게 아니라 내부 스크롤 영역만 스크롤됨)
- 상단 24px: 상태바 (좌측 시간, 우측 신호·와이파이·배터리 아이콘) — 실제 구현 시 OS가 제공하는 상태바이므로 앱에서 직접 그리지 않음, 와이어프레임에서는 레이아웃 확인용으로만 표시
- 하단 56px: 탭바 영역, 콘텐츠 배치 금지 (모든 화면 56px로 통일 확정)
- 좌우 최소 16px: 콘텐츠 여백

**v1 포함**: 회의 목록·생성(캘린더 기반 기간 선택)·슬롯 응답(필수용)·압축 후보 확인(선택용, 5개)·안건 등록(첨부·필독 포함)·회의 대시보드·완화요청 응답·수동 조정·참석 재확인(대리참석·온라인전환 포함)·회의 상세·알림 목록, `USER`의 부서·직급·직책·연락처, 재조율 자동화(1회 자동 + 2회 수동), 필수→선택 단계적 응답수집(조기 진행 포함), 참석 형태(대면/온라인/무관) 사전 선택 및 회의 모드 자동 계산, **회의실 자동 예약**(원래 v2였으나 모드 자동계산 흐름에 필수적이라 승격), **회의록·향후 추진 과제**(`MEETING_NOTE`, `ACTION_ITEM`, AI 초안 생성 포함 — 원래 v2였으나 통합 재검토 단계에서 v1로 승격, §10 참고)

**v2로 연기**: 없음 (회의록·향후 추진 과제도 v1로 승격 완료)

**확정된 세부 규칙**
- 후보 시작일은 생성일로부터 최소 3영업일 이후여야 함 (캘린더에서 이보다 이른 날짜 선택 불가)
- 선택 참석자가 압축 후보 5개 전부에 "불가"로 답하면 추가 장치 없이 바로 `confirmation_status`='불참'으로 기록

> 현재 v2로 연기된 항목은 없음(회의록·향후 추진 과제까지 전부 v1 승격 완료) — 데스크톱/태블릿 레이아웃 등 향후 확장 검토 항목이 생기면 별도 문서로 관리 예정

## 1. 회의 목록

**목적**: 내가 주최했거나 초대받은 회의를 한눈에 확인

**접근 대상**: 모든 로그인 사용자

**표시 데이터**
- 회의 제목 (`MEETING.title`)
- 상태 배지 (`MEETING.status`: 제안중 / 확정 / 재조율중 / 취소), 제안중이면 세부 단계(`MEETING.stage`)도 함께. **단, 확정됐지만 아직 참석 재확인이 완료되지 않은 회의는 "재조율중" 배지가 아니라 별도의 "재확인 대기"(amber) 배지로 구분 표시** — 재조율중(red)은 실제로 시간을 다시 조율하는 중일 때만 사용하고, 재확인 대기(amber)는 확정된 시간에 대해 참석 여부만 다시 확인받아야 하는 상태를 가리킴. 두 배지는 클릭 시 이동하는 화면도 다름(재확인 대기 → 화면 8, 재조율중 → 회의 대시보드/수동 조정)
- 회의록이 등록된 회의(`MEETING_NOTE.status`='등록')는 배지 라인 끝에 "회의록" 배지(문서 아이콘) 추가 표시 — 클릭 시 다른 라우팅 규칙보다 우선해 항상 회의 상세(화면 9)로 이동
- 내 역할 (`PARTICIPANT.role`: 필수 / 선택 / 주최자)
- 확정된 경우 확정 시간 (`TIME_SLOT.start_time`, `confirmed_slot_id` 참조)
- 내 응답 여부 배지 (미응답 시 강조 표시)

**액션**
- "새 회의 만들기" → 회의 생성 화면
- 항목 클릭 시 상태·역할에 따라 분기
  - stage='필수응답중' & 내 role=필수 & 미응답 → 슬롯 응답 화면(3)
  - stage='필수응답중' & 내 role=선택 → "대기 중" 표시만, 클릭 비활성
  - stage='선택확인중' & 내 role=선택 & 미응답 → 압축 후보 확인 화면(12)
  - 확정 & 재확인 미완료 → 참석 재확인 화면
  - 확정 & 재확인 완료 → 회의 상세 화면

**정렬/필터**: 마감 임박순 기본 정렬, 상태별 필터(**조율중**/확정/종료) — 필터 탭 명칭은 실제 구현 기준인 "조율중"으로 통일(이전 버전 문서의 "진행중" 표기는 오기)

## 2. 회의 생성

**목적**: 새 회의의 기본 조건 설정

**접근 대상**: 주최자(생성자)

**입력 필드**
- 제목 (`MEETING.title`)
- 참석자 — 이름 검색(드롭다운, 빠른 검색) 또는 검색창 옆 조직도 아이콘 클릭 시 뜨는 **부서별 조직도 모달**에서 선택 (`PARTICIPANT.user_id`). 부서가 많은 조직에서도 기본 화면이 길어지지 않도록 조직도는 별도 모달로 분리. 선택 직후 **필수 참석자 / 선택 참석자 여부를 명시적으로 지정**해야 함 (기본값 없음, 미지정 상태로 저장 불가). "추가된 참석자" 목록은 검색 영역과 구분된 별도 섹션으로 표시. 주최자 본인은 자동으로 필수 참석자로 추가되며 목록에서 제거 불가
- 회의 후보 기간 — **캘린더를 띄워 시작일·종료일을 직접 선택** (`MEETING.candidate_start_date`, `candidate_end_date`). 응답수집 기간(필수·선택 마감)은 이 기간을 기준으로 생성일부터 후보시작일 전날까지의 영업일 수에 비례해 자동 역산되므로 별도 입력 없음
- 회의 시간 길이 (`MEETING.duration_minutes`) — **주최자가 시간·분을 직접 입력하는 가변값으로 확정**(과거 "기본 1시간 고정값" 방침을 폐기). 최대 8시간, 분 단위는 5분 단위로만 입력 가능(둘 다 입력창에서 실제로 검증·제한됨 — 단순히 ">0"만 체크하던 이전 구현의 검증 누락 버그 수정 완료). `TIME_SLOT`은 더 이상 "고정 1시간 구간"이 아니라 이 `duration_minutes` 값을 기준으로 한 구간을 의미함(상세는 `db_model_mvp.md` 참고)
- 안건 마감 (`MEETING.agenda_deadline`) — 마감 날짜 선택 캘린더는 과거 날짜 선택 불가. 마감일이 오늘인 경우, 시(時)·분 선택 드롭다운도 현재 시각보다 이전 값은 선택 불가하도록 필터링

**입력하지 않는 것 (자동/후속 처리)**
- 온/오프라인 모드 — 주최자가 미리 정하지 않음. 참석자들이 슬롯 응답 시작 전 고르는 참석 형태(대면/온라인/무관)를 집계해 자동 계산 (화면 3·12 참고)
- 장소·화상회의 링크 — 모드가 계산되고 시간이 확정된 뒤, 회의실이 자동 매칭되거나 화상링크가 자동 생성됨 (화면 5 참고)

**액션**
- "회의 만들기" 클릭 시 확인 팝업 노출 (제목·참석자 수·후보 기간 요약 + "참석자 전원에게 알림이 발송됩니다" 안내) → 확인해야 최종 저장
- 저장 시 `required_response_deadline`, `optional_response_deadline`을 후보 시작일 기준 역산해 자동 계산
- 회의 후보 `TIME_SLOT`을 `candidate_start_date`~`candidate_end_date` 사이 30분 단위로 일괄 생성(슬롯 생성 간격은 30분 단위 그대로 유지 — 회의 자체의 길이(`duration_minutes`)와는 별개 개념)
- `MEETING.stage`='필수응답중'으로 시작. 필수 참석자 전원에게 초대 `NOTIFICATION` 즉시 발송, 선택 참석자에게는 "초대됨(대기 중)" 알림만 발송하고 실제 응답 요청은 단계가 '선택확인중'으로 넘어갈 때 별도 발송
- 저장 완료 시 주최자를 회의 대시보드(화면 5)로 자동 이동

**검증 규칙**
- 제목 필수
- 참석자(주최자 외) 최소 1명 — 주최자는 항상 자동으로 필수 참석자에 포함되므로 "필수 참석자 수"가 아니라 "전체 참석자 수"로 검증
- 추가된 참석자 전원 필수/선택 여부 선택 완료 (미지정 참석자가 있으면 저장 불가)
- 캘린더에서 시작일·종료일 모두 선택 완료 (종료일 ≥ 시작일, 시작일은 생성일로부터 최소 3영업일 이후만 선택 가능 — 그 이전 날짜는 캘린더에서 비활성 처리)
- **선택된 후보 기간에 평일이 최소 1일 이상 포함되어야 함** — 후보 기간이 주말로만 구성되면(예: 토~일만 선택) 슬롯 응답 화면(화면 3)의 시간 그리드가 평일만 생성하므로 실제 응답 가능한 `TIME_SLOT`이 0개가 되는 문제가 있었음. 주말로만 구성된 기간을 선택하면 안내 문구("선택한 기간에 평일이 없어 응답 가능한 시간대가 없어요. 평일을 포함해 다시 선택해주세요")를 표시하고 진행 차단
- 회의 시간 길이 입력값 검증(최대 8시간, 5분 단위) 통과
- 위 조건을 모두 충족하기 전까지 "회의 만들기" 버튼은 비활성화 상태

## 3. 슬롯 응답 (필수 참석자용)

**목적**: 필수 참석자가 참석 형태와 후보 시간대별 가능 여부를 표시해 후보를 압축하고 회의 모드를 정하는 데 쓰일 데이터를 제공

**접근 대상**: `PARTICIPANT.role`='필수'인 참가자(주최자 포함). 선택 참석자는 이 화면 대신 화면 12(압축 후보 확인) 이용

**표시 데이터**
- 내 정보: 원형 아바타(사진 또는 이니셜) + 이름, 부서·직급·직책 (`USER`), 그 아래 필수 배지 — 화면 우측 상단
- 회의 후보 기간의 날짜×시간 그리드, **30분 단위** (`TIME_SLOT` 목록). 열 헤더에 요일과 날짜를 함께 표시 (예: "목 7/10")
- **참석 형태 선택** (시간대 그리드 **아래쪽**, 회의당 1회) — "대면만 / 온라인만 / 둘 다 가능" 중 선택 (`PARTICIPANT.attendance_mode`: 대면/온라인/무관). **화면 순서를 "슬롯(일정) 먼저 선택 → 참석 형태 선택"으로 확정**(과거엔 참석형태를 그리드 진입 전에 먼저 물었으나, 일정 가능 여부를 먼저 파악하는 게 자연스럽다는 판단으로 순서 변경)
- 슬롯별 내 상태 (기본값 "가능" — 빈칸이 아니라 옅은 체크(✓)로 명시 표시해 "응답 안 함"과 구분되게 함)
- 필수 응답 마감까지 남은 시간 (`required_response_deadline` 기준, 상한선임을 안내)
- **안내 문구**: "이 응답은 회의 확정에 사용됩니다. 확정 후에는 일정 변경이 어려울 수 있으니 신중하게 답변해 주세요"

**응답 수정 및 재계산 규칙**
- 슬롯 응답과 참석 형태 모두 **마감 전까지는 언제든 자유롭게 수정 가능**("응답 완료" 이후에도 잠그지 않음)
- 단, 참석 형태를 응답 완료 이후에 변경하면 **회의 형태(`MEETING.mode`) 재계산이 트리거됨**을 사용자에게 토스트로 안내("참석형태가 변경되어 회의 형태가 다시 계산돼요")하고, 실제로 재계산을 수행
- **필수 참석자가 마감 시각까지 참석 형태를 응답하지 않으면 시스템이 자동으로 "무관"으로 간주**하고 회의 형태 계산을 진행(응답 누락으로 전체 진행이 막히지 않도록)

**액션**
- 슬롯 셀 클릭 → 가능 → 기피 → 불가 순환 토글 (`SLOT_RESPONSE` 생성/수정), 클릭 즉시 저장되며 별도 제출 버튼 없음. 클릭마다 짧은 "저장됨" 토스트로 피드백 제공
- **"응답 완료" 버튼** — 클릭 시 확인 팝업 노출("응답을 완료하시겠습니까? 이후 변경은 가능하지만, 회의가 이미 확정된 뒤에는 다른 참석자에게 영향을 줄 수 있습니다") → 확인 시 `PARTICIPANT.responded_at` 기록. 그리드 저장과는 별개 동작("다 확인했다"는 명시적 신호). 필수 참석자 전원이 응답 완료하면 `required_response_deadline` 전이라도 즉시 다음 단계로 조기 진행:
  1. 상위 5개 `TIME_SLOT`을 `is_shortlisted=true`로 압축
  2. 필수 참석자 전원의 `attendance_mode` 집계로 `MEETING.mode` 자동 계산 — **'온라인'을 선택한 사람이 한 명도 없으면 '대면'과 '무관'이 섞여 있어도 오프라인**, 온라인 선택자가 있고 대면 선택자가 없으면 온라인, 온라인·대면 선택자가 모두 있으면 하이브리드 (전원 무관도 온라인 선택자 0명이므로 자연히 오프라인) — 상세 근거는 `db_model_mvp.md`의 "회의 모드 자동 계산" 참고
  3. `MEETING.stage`='선택확인중'으로 전환
- 마감까지 자동 리마인드 (미응답 슬롯 대상)

## 12. 압축 후보 확인 (선택 참석자용)

**목적**: 필수 참석자 응답으로 압축된 소수의 후보 시간에 대해서만 선택 참석자의 참석 가능 여부를 가볍게 확인 (전체 그리드 아님)

**접근 대상**: `PARTICIPANT.role`='선택'인 참가자, `MEETING.stage`='선택확인중'일 때만 접근 가능 (그 전에는 회의 목록에 "대기 중"으로만 표시)

**표시 데이터**
- 내 정보: 화면 3과 동일한 상단 구성, 배지만 "선택 참석자"
- **참석 형태 선택** (화면 3과 동일한 컴포넌트, 회의당 1회) — 압축 후보 확인 전에 먼저 선택
- 이미 계산된 `MEETING.mode` 표시 (참고용 — 필수 참석자 기준으로 이미 정해졌음을 안내)
- 압축된 후보 목록 (`TIME_SLOT.is_shortlisted`=true, **5개**) — 각각 날짜·시간을 카드로 표시
- 선택 응답 마감까지 남은 시간 (`optional_response_deadline` 기준, 상한선임을 안내)

**프라이버시 안내**: "응답 내용은 개별 공개되지 않고, 집계 결과만 전달돼요. 단, 일정 조율이 어려운 경우 시스템이 필요한 분께만 개별로 변경 요청을 보낼 수 있어요." (완화요청 응답 화면(화면 6)에서 개인별로 이름·문구가 노출되는 실제 동작과 정합되도록 caveat 문장 추가)

**액션**
- 각 후보 카드에 **가능 / 불가** 2択만 표시 (기피 없음 — 참석 여부만 가볍게 확인하는 목적이라 세밀한 선호 표시는 불필요)
- **"응답 완료" 버튼** — 클릭 시 확인 팝업 노출("응답을 완료하시겠습니까?") → 확인 시 `PARTICIPANT.responded_at` 기록. 선택 참석자 전원이 응답 완료하면 `optional_response_deadline` 전이라도 즉시 `MEETING.stage`='확정'으로 전환
- 압축된 후보 **5개 전부에 "불가"**로 응답하면 완화요청·재조율을 트리거하지 않고 `confirmation_status`='불참'으로만 기록, 나머지 진행(필수 참석자 기준 확정)은 그대로 진행 — 상세는 아래 "선택 참석자 불참 처리 후속 안내" 참고

### 선택 참석자 불참 처리 후속 안내

압축 후보 5개 전부에 "불가"로 응답해 `confirmation_status`='불참'으로 기록된 선택 참석자에게는:
- "회의 시작부터 회의록을 확인할 수 있어요" 안내 메시지를 표시(v1 기준 — 회의록·향후 추진 과제 화면(화면 10)이 v1로 승격되어 실제 이동 가능한 링크/버튼으로 안내)
- 같은 내용으로 `NOTIFICATION` 생성

이 후속 안내는 화면 9(회의 상세)·화면 12(이 화면) 양쪽에서 참조하는 공통 규칙이다.

## 4. 안건 등록

**목적**: 회의 안건 공유

**접근 대상**: 모든 참가자(주최자 포함) — 안건 마감 시각과 무관하게 항상 접근 가능(마감 이후에도 등록·수정·삭제 가능하도록 열어둠. 마감은 리마인드 발송 기준일 뿐 접근을 막는 기준이 아님 — 이전 버전 문서의 "안건 마감 전까지" 표기는 실제 동작과 불일치했던 오기)

**입력 필드**
- 안건 제목 (`AGENDA.title`)
- 설명 (`AGENDA.description`)
- 필독 여부 (`AGENDA.is_required`)
- 첨부파일 (`AGENDA.attachment_url`) — 첨부 시 파일명과 삭제(x) 버튼이 표시되고, "파일 선택"(다른 파일 선택) 버튼은 그 아래로 위치 이동

**액션**
- 안건 추가 / 수정 / 삭제 — 수정·삭제는 **작성자 본인만 가능**(순서 변경 기능은 검토 후 스펙에서 제외하기로 확정 — 등록순으로만 표시) (`AGENDA.author_id`로 서버에서도 검증). 등록·수정·삭제 클릭 시 확인 팝업 노출(삭제는 "이 안건을 삭제하시겠습니까?" 등 되돌릴 수 없음을 강조)
- 안건 등록·수정·삭제 시 **작성자 본인을 제외한** 나머지 참가자에게 `NOTIFICATION` 생성 → 탭바 "알림" 아이콘에 빨간 점 배지로 표시
- 마감 경과 시 주최자에게 리마인드만 발송 (진행에는 영향 없음)
- "안건 등록"/"수정 완료" 버튼은 제목이 입력되기 전까지 비활성화 상태

**표시**: 등록된 안건 리스트 (필독 우선 정렬), 카드마다 작성자 원형 아바타 + 이름 표시 (`AGENDA.author_id` → `USER`, 등록한 본인의 id로 자동 설정)

## 5. 회의 대시보드 (주최자 전용)

**목적**: 응답 현황과 후보 시간 순위 확인, 회의실·모드 자동 처리 결과 확인, 필요 시 개입

**표시 데이터**
- 현재 진행 단계 표시 (`MEETING.stage`: 필수응답중 / 선택확인중 / 확정)
- 참가자별 응답 상태 (응답완료 `responded_at` 유무 / 미응답) — 필수·선택 단계별로 구분 표시
- 후보 시간별 하드 제약 통과 여부 · 소프트 선호 점수 (필수응답중 단계에서 계산)
- 압축된 후보 목록(5개) 및 선택 참석자 응답 현황 (선택확인중 단계 이후)
- 계산된 회의 모드(`MEETING.mode`) — 필수 참석자 `attendance_mode` 집계 결과
- 확정 시: 자동 매칭된 회의실(`ROOM_BOOKING` → `ROOM.name`) 또는 자동 생성된 화상링크 — **주최자 승인 대기 상태**로 표시
- 재조율 횟수 (`MEETING.reschedule_count`) — **자동 1회 + 수동 2회, 총 3회 한도**를 실제로 강제. 1회차는 자동, 2·3회차는 수동으로 표기. 한도(3회)에 도달하면 더 이상 재조율을 진행하지 않고 담당자 에스컬레이션 안내("재조율 한도(자동 1회 + 수동 2회)에 도달했어요. 담당자에게 문의해주세요")로 전환
- `PARTICIPANT.confirmation_status`='확정'은 **회의 시간이 최종 확정되는 순간 시스템이 전체 확정 참석자에게 자동으로 기록**(화면 9의 "참석으로 변경"은 그 이후의 예외적 재전환 경로로 별도 유지)

**액션**
- 추천 순위 확인 후 수동으로 시간 확정 가능 — **최종 확정 슬롯 선택 알고리즘은 점수 기준(하드 제약 통과 후보 중 `unavailCount` 최소)으로 통일**. "강제 진행" 버튼도 동일 알고리즘을 사용(배열 순서상 첫 하드패스 후보를 고르던 과거 방식은 폐기)
- 완화요청 발송 상태 확인 — 완화요청 순차 알고리즘은 **사람 기준**(한 슬롯에 대해 후보자에게 순서대로 요청)으로 전체 화면(화면 6·7) 통일. 요청 큐/이력은 신설된 `MITIGATION_REQUEST` 엔티티(`meeting_id`, `slot_id`, `target_user_id`, `status`, `requested_at`)로 저장
- 재조율 2·3회차(수동) 도달 시 "수동 조정" 화면으로 이동하는 버튼 노출, 한도(3회) 도달 시에는 수동 조정 대신 담당자 에스컬레이션 안내로 대체
- 단계 조기 진행 여부와 관계없이, 주최자가 원하면 수동으로 다음 단계로 강제 진행하는 버튼 제공
- **회의실/링크 자동 결과 승인 또는 변경** — 확정 시 시스템이 자동 매칭한 회의실이나 생성한 화상링크를 주최자가 확인하고 그대로 승인하거나, 다른 회의실로 직접 변경 가능. 적합한 회의실이 없었던 경우 여기서 "온라인으로 전환" 또는 재시도 선택

## 6. 완화요청 응답 (조건부 노출)

**목적**: 하드 제약을 건 당사자에게 완화 의사 확인

**노출 조건**: 필수 참석자 전원의 하드 제약을 만족하는 슬롯이 없을 때, 제약을 건 당사자에게 순차 노출

**표시 데이터**
- 문제가 되는 후보 시간
- 본인이 표시했던 제약 내용 (요일/시간대, 하드 여부)

**액션**
- 변경 요청에 동의("가능으로 변경") → 재평가 트리거. 이 액션만 데이터 변경이 있으므로 확인 팝업 노출
- "불가로 유지"(데이터 변경 없음) → 다음 당사자에게 순차 요청(사람 기준 순차 — 화면 7과 통일된 알고리즘), 모두 유지 시 주최자에게 알림. 데이터 변경이 없는 액션이므로 **확인 팝업 없이 즉시 처리**(전역 확인 팝업 규칙: 데이터를 실제로 바꾸는 액션에만 팝업 적용)

---

## 7. 수동 조정 (주최자 전용, 조건부 노출)

**목적**: 자동화가 실패했을 때 주최자가 직접 결정

**노출 조건**
- 완화요청 대상 전원이 "불가로 유지"
- 확정 후 재조율이 2회차(수동 1회차)에 도달
- 회의실 자동 매칭 실패

**표시 데이터**: 지금까지 시도한 후보/불가 유지 이력 요약 (누가 몇 번째에 무엇을 불가로 유지했는지 — 사람 기준 순차 알고리즘, 화면 6과 통일)

**액션**
- 참석자 역할 변경 (필수 → 선택)
- 후보 기간 확장 후 재검색 — **연장 폭은 7일로 최종 확정**
- 시간 강제 확정 — 재조율 한도(자동 1회 + 수동 2회, 총 3회)에 도달하면 이 액션 대신 담당자 에스컬레이션 안내로 대체
- 회의실 재선택 또는 온라인 전환 (자동 매칭 실패 시) — "회의실 재선택" 버튼은 실제 동작하도록 구현 완료(이전에는 타이틀만 있고 클릭 핸들러가 없던 버그 수정)

---

## 8. 참석 재확인

**목적**: 확정된 시간에 대한 최종 참석 의사 확인

**접근 대상**: 모든 참가자, 최초 확정 또는 재조율에 의한 재확정 시마다 노출

**표시 데이터**
- 확정 시간 · 장소(자동 매칭된 회의실) · 화상링크
- 재조율로 변경된 경우, 이전 시간 대비 변경 안내 문구
- 슬롯 응답 시 미리 선택했던 내 참석 형태(`attendance_mode`) 표시 — 그사이 사정이 바뀌었을 수 있어 재확인 성격

**액션** (세 옵션 동시 노출, 아래 우선순위로 시각적 강조 차등)
1. **참석 확정 — 온라인 전환** (기본/강조 버튼): 참석자는 그대로 유지되므로 재조율 불필요. `PARTICIPANT.attendance_mode`="온라인"으로 갱신
2. **대리 참석자 지정** (보조 버튼): 온라인 참여도 불가능할 때 선택. `PARTICIPANT.delegate_user_id` 설정
3. **불참 통보** (약하게 표시, 최후 수단): 클릭 시 **사유 입력(자유 텍스트, 필수)** 을 먼저 요구 — 필수 참석자인 경우 입력창 위에 "다른 참석자들이 이미 이 일정에 맞춰 계획을 세웠을 수 있습니다. 재조율이 발생하면 일정이 빠듯해질 수 있어요"라는 경고를 추가로 표시 → 제출 시 `PARTICIPANT.decline_reason` 저장 및 `confirmation_status`='불참'. 이후 처리는 역할에 따라 분기:
   - **선택 참석자**: 기록만 하고 진행은 그대로. "회의 후 회의록을 확인해주세요" 안내 메시지 표시(v2 회의록 기능 완성 전까지는 안내 문구만, 실제 이동 링크는 v2에서 완성) + 같은 내용으로 `NOTIFICATION` 생성
   - **필수 참석자**: 재조율 프로세스 트리거 (기존 규칙 그대로). "불참이 접수되었습니다. 다른 시간으로 재조율을 시작합니다" 안내 메시지 표시 — 회의록 안내는 표시하지 않음(아직 시간이 재조율 중이라 회의가 끝난 게 아님)

**설계 노트**: 세 옵션을 단계별로 숨기지 않고 한 화면에 동시 노출한다. 다만 "온라인 전환"을 사람이 그대로 유지되는 가장 손실 적은 대안으로 보고 시각적으로 가장 강조한다. 회의가 오프라인으로 확정되어 `MEETING.video_link`가 없었더라도, 온라인 전환 선택 시 화상링크를 자동 생성한다.

## 9. 회의 상세

**목적**: 확정된 회의의 전체 정보 확인 (읽기 전용)

**표시 데이터**
- 확정 시간 · 장소(`ROOM_BOOKING` → `ROOM.name`, `ROOM.location`) · 화상링크
- 참석자 목록 및 각자의 확정 여부, 참석 형태(대면/온라인)
- 안건 목록

**액션**
- 캘린더에 추가
- **회의가 시작된 시점부터**(진행 중·종료 공통, 확정/불참 참석 상태와 무관하게) "회의록 확인하기" 버튼이 활성화되어 회의록·향후 추진 과제 화면(화면 10)으로 이동 가능(v1 — 과거엔 "종료 후"·v2 예정이었으나 통합 재검토에서 승격 확정)
- 참석 상태가 확정인 참석자 중 압축 후보 전부에 불가/불참 처리되어 있었던 경우, 회의 시작 전이면 "참석으로 변경" 가능 — 이때 `confirmation_status`가 '확정'으로 재전환됨(정상 확정 경로는 회의 시간이 최종 확정되는 순간 시스템이 자동 기록하며, 이 버튼은 그 예외적 재전환 경로로 별도 유지)

## 10. 회의록 · 향후 추진 과제 (v1로 승격)

**목적**: 회의 내용 기록 및 후속 실행 항목 추적. 원래 v2 예정이었으나 통합 재검토 과정에서 v1로 승격, 자체 설계로 제작 완료(`wireframe_17_meeting_minutes_1.html`, frame-label "화면 10")

**접근 대상**: 회의 참가자 전원(열람은 누구나 가능). 유일한 진입 경로는 회의 상세(화면 9)의 "회의록 확인하기" 버튼 — 별도의 독립 목록/탭 없음

**접근 가능 시점**: **회의가 시작된 시점부터**(진행 중·종료 공통) — 실시간 작성 가능. "회의 종료 후"로 제한하지 않음

**작성 권한**
- 작성자는 기본값 **주최자**이며, **주최자가 참석자 중 다른 사람을 작성자로 재지정 가능**("변경" 버튼, 주최자 로그인 시에만 노출)
- **편집 권한은 오직 현재 지정된 작성자 1인**에게만 있음. 작성자가 아닌 사용자는 "회의 내용" 텍스트 영역이 읽기 전용으로 바뀌고 "이 회의록은 {작성자}님만 작성할 수 있어요" 안내 노출, AI 초안 생성·저장 버튼도 숨김(열람은 항상 가능)
- **향후 추진 과제는 이 권한 제한과 무관하게 참석자 누구나 추가·완료 체크 가능**

**회의록 양식 필드** (자동 채움, 읽기 전용): 날짜·시간(`MEETING`), 참석인원(`PARTICIPANT`, **직위 순으로 나열, 동일 직위는 이름 가나다순** — 예: "과장 김민준, 대리 박서연, 대리 최유나…"), 안건 제목(`AGENDA`, 첨부파일이 있으면 파일명 칩 표시). 작성자가 실제로 입력하는 곳은 그 아래 "회의 내용" 텍스트 영역뿐

**저장 방식**: "임시저장" / "등록" 2단계
- 등록 전까지는 자유롭게 임시저장(알림 없음, 언제든 재열람·재편집 가능)
- **"등록" 시점에만 해당 회의 참석 인원 전체에게 "회의록 등록" 안내 알림**(`NOTIFICATION.type`='회의록등록') 발송
- 등록 후에는 "저장" 버튼 하나로 계속 편집 가능하나 재알림은 없음

**AI 초안 생성**: 녹음/STT 기반이 아니라, 작성자가 "회의 내용" 칸에 입력해둔 러프한 메모를 원본으로 LLM(예: Claude API) 호출을 통해 다듬어진 문단으로 재정리해 같은 칸에 덮어쓰는 방식. 자동 채움 필드(날짜·시간·참석인원·안건)는 AI가 건드리지 않음. "회의 내용"이 비어있으면 버튼 비활성화

**향후 추진 과제** (`ACTION_ITEM`, 화면 표시 용어 — 내부 데이터 모델 명칭은 `ACTION_ITEM` 유지): 목록(체크박스 완료 처리, 담당자 칩, 마감일 칩 — 마감 지났고 미완료면 강조), "+ 항목 추가" 인라인 폼(내용·담당자·마감일 — 마감일은 화면 2·17과 동일한 커스텀 캘린더 컴포넌트 재사용), 개별 삭제

**회의 목록(화면 1) 연동**: `MEETING_NOTE.status`='등록'이 되면 화면 1의 해당 회의 카드에 "회의록" 배지 표시, 클릭 시 다른 라우팅 규칙보다 우선해 항상 회의 상세(화면 9)로 이동 → 화면 9의 "회의록 확인하기"로 이어짐

**데이터 모델**: `MEETING_NOTE`(meeting_id, content, author_user_id, is_ai_generated, status: 임시저장/등록, registered_at, updated_at), `ACTION_ITEM`(id, meeting_id, content, assignee_user_id, due_date nullable, is_done, created_by, created_at) — `db_model_mvp.md` 반영 완료

## 11. 알림 목록

**목적**: 내가 참여 중인 회의들에서 발생한 변경사항을 한곳에서 확인

**접근 대상**: 모든 로그인 사용자 (탭바 "알림" 탭)

**표시 데이터**
- `NOTIFICATION` 목록 (최신순), 항목마다 관련 회의 제목·알림 문구·발생 시각
- 미확인(`is_read=false`) 항목은 강조 표시(굵게 또는 좌측 점)

**액션**
- 항목 탭 → 알림 유형·현재 로그인 사용자의 역할에 따라 아래처럼 분기 이동, 해당 항목 `is_read=true`로 변경
  - "참석재확인" → **참석 재확인 화면(화면 8)**(과거 화면 9로 잘못 라우팅되던 버그 수정 완료)
  - "응답요청"·"마감임박" → **참석자 역할에 따라 분기**: 필수 참석자면 슬롯 응답 화면(화면 3), 선택 참석자면 압축 후보 확인 화면(화면 12)로 이동. 주최자에게 온 경우에만 회의 대시보드(화면 5)로 이동(화면 5는 주최자 전용이라 참석자에게는 애초에 부적절한 목적지였던 문제 수정)
  - 그 외 유형(시간확정/안건등록/안건수정/재조율/불참안내/완화요청/회의실확정/회의록등록 등) → 기존과 동일하게 관련 회의(대시보드 또는 상세)로 이동
- 알림 타입 태그는 내부 시스템 용어를 그대로 노출하지 않음 — "완화요청"(내부 `NOTIFICATION.type` 값)은 화면에는 "변경요청"으로 표시(데이터 값 자체는 유지, 표시 라벨만 매핑)
- 탭바 "알림" 아이콘: `is_read=false` 항목이 1건 이상이면 빨간 점 배지 표시 (개수 아닌 존재 여부만)
