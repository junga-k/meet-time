구현 노트 (Claude Code 참고용)

  • 노출 조건: 필수 참석자 전원의 하드 제약(누군가 "불가"로 표시한 슬롯은 제외)을 만족하는 후보 시간이 하나도 없을 때, 그 하드 제약을 건 당사자에게 트리거됨. 여러 명의 하드 제약이 겹쳐 문제라면 한 번에 한 명씩 순차로 요청(스펙 "제약을 건 당사자에게 순차 노출") — 이 화면은 "지금 요청 대상인 한 사람"의 화면만 보여주며, 대기열 관리는 서버 로직(이 와이어프레임엔 미표현)
  • 선택 참석자에게는 이 화면의 대응 화면이 없음: 이 흐름은 필수 참석자의 전체 그리드 응답(화면 3)에서 나온 하드 제약(불가)에만 적용됨 — 압축된 5개 후보는 애초에 필수 참석자 하드 제약을 전부 통과한 시간만 올라오므로, 선택 참석자가 보는 화면 12에는 "하드 제약" 개념 자체가 없음(가능/불가 2択만 존재). 선택 참석자가 5개 전부 "불가"로 답해도 이 화면처럼 다시 물어보지 않고 바로 confirmation_status='불참'으로 기록하고 필수 참석자 기준으로 회의를 그대로 진행함(wireframe_04_shortlist_confirm.html의 "5개 전부 불가" 분기 참고) — project_handoff.md에 기록된 설계 근거: 참석형태(attendance_mode) 사전선택으로 이미 모드 불일치 문제가 해소됐다고 보고, 선택 참석자까지 조율 협상에 끌어들이는 추가 장치를 의도적으로 생략함
  • 진입 경로: NOTIFICATION(type: 완화요청 — db_model_mvp.md의 시스템 용어) 탭 → 이 화면. 화면 11(알림 목록) 완성 후 실제 연결 예정. 화면 안 사용자 문구는 "완화" 대신 "변경"으로 통일했음(이지훈님 입장에서는 자기 응답을 "바꿔주는" 행동이지 "기준을 낮춰주는" 느낌의 "완화"라는 단어가 와닿지 않는다고 판단 — 버튼도 이미 "가능으로 변경"이라 문구였어서 나머지 문구도 여기 맞춤). 스펙·데이터 모델상의 공식 용어(완화요청)와 화면 문구(변경)가 달라진 것이므로 실제 구현 시 참고
  • "변경" 동의 시 반영값: 스펙엔 "완화 동의 → 재평가 트리거"라고만 되어 있고 SLOT_RESPONSE.status가 정확히 무엇으로 바뀌는지는 명시돼 있지 않음 — 이 와이어프레임은 불가 → 가능으로 전면 전환하는 것으로 가정함(기피로 부분 변경하는 옵션은 없음). 부분 변경(기피)까지 지원하려면 버튼을 3지선다로 확장하는 방안을 검토 필요
  • 거절 시: 이 참석자의 응답은 "불가"로 그대로 유지되고, 시스템이 다음 하드 제약 보유자에게 순차로 요청을 이어감. 모두 거절하면 주최자에게 알림이 가고 화면 5(대시보드)에 "완화요청 발송 상태"로 반영됨(대시보드 와이어프레임에 이미 구현된 배너 — 대시보드는 주최자 시점이라 시스템 용어를 그대로 사용)
  • 주요 액션 확인 팝업 대상 목록(screen_specs.md 공통 규칙)에 이 화면의 두 액션이 명시적으로 나열돼 있진 않지만, 응답을 실제로 바꾸는 되돌리기 어려운 결정이라 다른 화면과 동일한 확인 팝업 패턴을 적용함
  • 응답 후에는 액션 버튼이 사라지고 완료 상태 화면으로 바뀜 — 동의 시 "재평가 중" 안내, 거절 시 "다른 분께 요청을 이어간다"는 안내만 표시(본인이 마지막 순번인지 여부는 알 수 없으므로 그 정보는 노출하지 않음)
  • 이 와이어프레임은 문제 후보가 1개인 경우를 시연. 동시에 여러 후보가 각각 다른 사람의 하드 제약으로 막혀 있는 경우, 서버가 우선순위를 정해 이 화면 패턴을 슬롯별로 순차 반복 노출할 것으로 가정(스펙에 세부 규칙 없음)
