구현 노트 (Claude Code 참고용)

  • 목록 항목 = 로그인 사용자가 PARTICIPANT로 속한 MEETING 전체(주최자 포함). 정렬은 마감 임박순이 기본값이나, 단계별로 "마감"의 의미가 다름: 필수응답중 → required_response_deadline, 선택확인중 → optional_response_deadline, 확정 이후 → 확정 시간(TIME_SLOT.start_time) 임박순으로 자연 전환. 종료/취소 건은 마감 개념이 없어 정렬 최하단 고정
  • 카드 내 태그(배지) 표시 순서는 역할 배지(주최자/필수/선택) → 상태 배지(제안중/재조율중/재확인대기/확정/종료·취소) → 세부 단계 배지(필수응답중/선택확인중) 순으로 왼쪽부터 배치
  • 필터 이름은 스펙 초안의 "진행중/확정/종료"에서 "조율중/확정/종료"로 변경 — 앱 성격이 "회의 진행" 관리가 아니라 "일정 조율"이라 조율중이 더 정확한 표현이라 판단. 매핑: 조율중=제안중+재조율중+(확정됐지만 참석 재확인 미완료인 건 포함), 확정=확정 & 재확인 완료, 종료=취소 건(회의 시작 시간이 지난 확정 건도 이후 종료로 분류 예정이나 이 와이어프레임 데모 데이터엔 미포함) — MEETING.status enum과 1:1 대응이 아니라 화면 표시용 파생 카테고리이므로 실제 구현 시 이 매핑 로직을 그대로 따를지 확인 필요
  • 카드 클릭 라우팅(스펙 화면 1 "액션" 그대로): ① stage=필수응답중 & 내 role=필수 & 미응답 → 슬롯 응답 화면(3). ② stage=필수응답중 & 내 role=선택 → 클릭 비활성, "대기 중" 표시만. ③ stage=선택확인중 & 내 role=선택 & 미응답 → 압축 후보 확인 화면(12). ④ 확정 & 재확인 미완료 → 참석 재확인 화면(8). ⑤ 확정 & 재확인 완료 → 회의 상세 화면(9). 이 와이어프레임은 단일 화면 목업이라 실제 페이지 이동 대신 토스트로 이동할 화면을 표시하는 것으로 대체 시연함
  • 이미 응답 완료(responded_at 존재)한 필수/선택 참석자가 같은 단계의 카드를 다시 클릭하면 — 스펙에 명시는 없으나, 화면 3·12 자체가 "응답 완료 후에도 값 수정 가능"하다고 되어 있으므로 동일하게 해당 응답 화면으로 재진입 가능하다고 가정(수정 목적)
  • "미응답" 배지는 아직 첫 응답(슬롯 응답·압축 후보 응답)을 안 한 경우에만 사용 — 필수/선택 응답 단계(제안중, 미응답)에만 노출. 확정된 회의는 이미 응답을 마친 상태이므로 "미응답"이라는 표현이 맞지 않아, 확정 & 참석 재확인 미완료인 경우는 상태 배지 자체를 "확정"이 아니라 별도의 앰버색 "재확인 대기" 배지(status-badge.reconfirm)로 바꿔 표시(하단 isPendingReconfirm()). (수정) 이전 버전에서는 이 케이스에 진짜 재조율중(완화요청 전원 거절 등으로 일정을 처음부터 다시 잡는 경우, status==="재조율중")과 동일한 빨간색 "재조율중" 배지를 재사용했으나, 두 상태는 발생 사유와 뒤이은 사용자 액션(참석 재확인 화면 8 vs 회의 대시보드·수동 조정 화면 5·7)이 서로 달라 혼동을 줄 수 있다는 판단하에 배지를 분리 — "재확인 대기"(앰버)는 status==="확정" && !reconfirmed일 때만, "재조율중"(빨강)은 status==="재조율중"일 때만 표시되도록 getStatusBadge()에서 구분
  • 주최자(role=주최자)는 필수 참석자와 동일한 응답 대상이지만, 회의가 재조율중이거나 자신이 개입해야 하는 상태(완화요청 전원 거절, 회의실 매칭 실패 등)일 때는 카드 클릭 시 회의 대시보드(화면 5) 또는 수동 조정(화면 7)으로 라우팅됨 — 이 화면에서는 데모 카드 1개로 "재조율중 → 대시보드" 케이스만 대표로 시연
  • "새 회의 만들기" 버튼은 역할과 무관하게 항상 노출 — 클릭 시 회의 생성 화면(2)으로 이동 (실제 구현에서는 로그인 사용자가 그대로 주최자로 자동 지정됨)
  • 상단 헤더는 특정 회의로 진입한 하위 화면들(화면 2·3·4·12)과 달리 탭 최상위 화면이라 뒤로가기 화살표 없음 — 탭바 "내 회의" 아이콘이 곧 이 화면의 진입점
  • (신규) 검색창 추가 — 필터 세그먼트 위에 회의 제목 인라인 검색만 추가(별도 검색 화면 없음). 필터와 검색어는 AND 조건으로 함께 적용됨. 회의록 검색은 화면 10(v2) 제작 시 별도로 결정하기로 함
  • (수정) 상단 카운트 라벨이 필터/검색과 무관하게 항상 전체 건수만 보여주던 기존 버그를 고쳐서, 필터나 검색어가 적용된 경우 결과 건수를 표시하도록 변경(전체 필터+검색어 없음일 때만 "총 N건" 유지)
  • (신규) "회의록" 배지 추가 — 해당 회의의 MEETING_NOTE가 등록(status=등록) 상태이면(화면10, `wireframe_17_meeting_minutes_1.html`의 "등록" 버튼으로 확정된 경우) 배지 라인 맨 끝에 파란색 "회의록" 배지(note-badge, 문서 아이콘)를 표시. 데모에서는 id=5 "팀 빌딩 워크샵 기획"(확정·재확인 완료 건)에 hasRegisteredNote: true를 부여해 시연. 이 배지가 붙은 회의를 클릭하면 항상 회의 상세(화면 9)로 이동하도록 라우팅을 고정(다른 라우팅 규칙보다 우선) — 화면9에 진입한 뒤 그 화면의 "회의록 확인하기" 버튼으로 화면10까지 이어지는 기존 흐름을 그대로 탄다. 실제 구현 시에는 hasRegisteredNote가 아니라 MEETING_NOTE.status === '등록' 존재 여부로 대체
