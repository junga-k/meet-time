구현 노트 (Claude Code 참고용)

  • 이 화면은 PARTICIPANT.role과 무관하게 MEETING.organizer_id === 현재 로그인 사용자일 때만 접근 가능(주최자 전용). 상단 [데모 전용 컨트롤] 박스는 MEETING.stage(필수응답중/선택확인중/확정)에 따라 화면 구성이 크게 달라지는 것을 한 파일로 보여주기 위한 와이어프레임 전용 장치이며 실제 화면에는 없음 — 실제로는 서버 상태값 하나로 결정됨
  • 참가자 응답 현황: 필수 참석자는 항상 표시, 선택 참석자 섹션은 stage가 선택확인중 이상으로 전환된 뒤부터 응답 현황이 채워짐(그 전엔 "필수 응답 완료 후 시작" 안내만). 각 참가자 responded_at 유무로 완료/미응답 표시
  • "리마인드" 버튼(신규): 마감 임박 시 시스템이 자동으로 보내는 리마인드(NOTIFICATION type: 마감임박)와는 별개로, 주최자가 원하는 시점에 미응답자 개별 또는 "미응답자 전체 리마인드" 버튼으로 한 번에 응답요청 알림(type: 응답요청, db_model_mvp.md에 추가함)을 즉시 보낼 수 있음. 이미 응답완료한 참석자에겐 노출 안 됨. 한 번 보내면 "전송됨"으로 바뀌고 재전송 버튼은 사라짐(중복 알림 방지) — 실제 구현에서는 일정 시간 경과 후 재전송 가능하게 할지 등 쿨다운 정책은 별도 결정 필요. 다른 주요 액션과 달리 되돌리기 쉬운 가벼운 액션이라 확인 팝업 없이 클릭 즉시 전송·토스트 피드백 처리. 데모 편의를 위해 리마인드를 보내면 일정 시간 후(1.8초~) 해당 참석자가 실제로 응답을 완료한 것처럼 자동 전환되도록 만들어, "전원 응답완료" 시 다음 단계로 넘어가는 화면까지 이 화면 하나에서 계속 지켜볼 수 있음 — 실제 서비스에서는 당연히 참석자가 직접 화면 3·12에서 응답해야 반영되는 것이며 리마인드를 보냈다고 자동으로 응답 처리되는 기능이 아님(데모 전용 시뮬레이션일 뿐)
  • 후보 시간 순위(필수응답중 단계): SLOT_RESPONSE 집계로 하드 제약(누군가 "불가"로 표시) 통과 여부와 소프트 선호 점수("가능"·"기피" 가중 평균, 스코어링 알고리즘은 서버 로직 별도 설계 필요 — 이 와이어프레임은 데모용 정적 점수만 표시)를 계산해 순위 표시. 주최자가 "이 시간으로 확정"을 누르면 필수 참석자 전원 응답을 기다리지 않고 즉시 확정 가능(확인 팝업 포함) — 조기 확정은 스펙의 "단계 조기 진행" 권한과 별개로, 주최자가 원하는 시점에 항상 강제 확정할 수 있는 액션으로 이 와이어프레임에서 추가함
  • 완화요청 상태 박스: 하드 제약을 통과하는 슬롯이 전혀 없을 때 노출된다고 가정(화면 6 완화요청 응답 트리거 조건과 연동). 스펙 화면 5는 "완화요청 발송 상태 확인"이라고만 되어 있고 정확한 노출 조건·문구는 명시돼 있지 않아 이 와이어프레임에서 위와 같이 가정함
  • 압축 후보 확인 현황(선택확인중 단계부터): TIME_SLOT.is_shortlisted=true인 5개 슬롯별로 선택 참석자들의 가능(초록)/불가(빨강) 상태를 점으로 표시하되, "응답완료" 버튼을 아직 안 누른 참석자는 점을 빈 테두리로 가려서 표시(값이 있어도 미노출). SLOT_RESPONSE는 화면 3과 동일하게 레코드가 없으면 기본값 "가능"으로 처리되므로(db_model_mvp.md, wireframe_04와 동일 규칙) 완료 전 참석자의 후보별 값을 그대로 보여주면 "기본값일 뿐인데 이미 응답한 것"처럼 보여 혼동을 줄 수 있음 — 그래서 완료한 참석자의 점만 노출해 상단 "응답완료 n/m" 숫자와 항상 일치하도록 정리함
  • 회의 형태(MEETING.mode)는 필수 참석자 전원 응답 완료 시점(선택확인중 전환)에 계산되므로 필수응답중 단계에서는 "계산 전" 상태로 표시, 그 이후 단계부터 계산된 값(오프라인/온라인/하이브리드)과 판단 근거 요약 문구 표시
  • 확정된 회의 시간 배너(신규): 확정 단계로 전환되면 화면 최상단에 "확정된 회의 시간"을 크게 강조 표시(MEETING.confirmed_slot_id → TIME_SLOT.start_time/end_time). 원래 스펙 화면 5의 "표시 데이터" 목록엔 회의실·모드·재조율 횟수는 있었지만 정작 몇 시로 확정됐는지 자체가 빠져 있어 — 어느 후보 시간으로 확정됐는지 한눈에 안 보이면 대시보드로서 의미가 없으므로 화면과 screen_specs.md 화면 5 "표시 데이터"에 함께 추가함
  • 회의실/화상링크 자동 매칭 결과는 확정 단계에서만 노출. "이대로 승인"/"다른 회의실로 변경" 두 액션 제공 — 변경 시 대안 회의실 목록을 하단 슬라이드업 팝업으로 노출(안건 등록 화면의 팝업과 동일한 인터랙션 패턴 재사용). 적합한 회의실이 없어 온라인 전환된 경우는 회의실 박스 대신 화상링크만 표시하는 것으로 가정(이 와이어프레임엔 미시연 — 데모 데이터는 회의실 매칭 성공 케이스만 포함)
  • 재조율 횟수(MEETING.reschedule_count)는 확정 단계 하단에 항상 표시. 2회 이상이면 "수동 조정 화면으로 이동" 버튼과 경고 문구가 추가로 노출됨(스펙 db_model_mvp.md "reschedule_count ≥ 2가 되면 자동 재조율 대신 주최자 수동 처리로 전환" 규칙 반영) — 상단 데모 컨트롤의 "재조율 횟수" 칩으로 0/1/2/3(한도) 케이스를 모두 확인 가능. "재조율 시도" 버튼(신규): 이 화면엔 원래 재조율 발생을 실제로 트리거하는 액션이 없어 reschedule_count가 항상 데모 컨트롤로만 세팅되던 문제가 있었음(open_decisions_tracker.md A4) — 재조율 이력 섹션에 실제로 카운트를 증가시키는 버튼을 추가하고, screen_specs.md "재조율 자동화(1회 자동 + 2회 수동)" 규칙대로 1회차는 자동, 2·3회차는 수동으로 구분 표시하며, 총 3회(자동 1회+수동 2회) 도달 시 더 이상 진행 불가 + "담당자에게 문의" 안내로 전환되도록 함(실제 백엔드 에스컬레이션 없이 UI 상태만 데모)
  • "다음 단계로 강제 진행" 버튼은 현재 단계 대상자가 전원 응답하지 않았어도 주최자가 원하면 즉시 다음 단계로 넘길 수 있는 액션(스펙 "단계 조기 진행 여부와 관계없이... 수동으로 다음 단계로 강제 진행하는 버튼 제공"). 확정 단계에서는 이 버튼이 사라지고 대신 회의실 승인 액션이 그 자리를 대신함. 앱 전체 공통 규칙(주요 액션 확인 팝업)에 따라 클릭 시 확인 팝업 노출 — 스펙의 확인 팝업 대상 목록에 이 액션이 명시적으로 나열되어 있진 않지만, 되돌리기 어려운 주최자 액션이라 동일 패턴을 적용함
  • 이 버튼과는 별개로, 참석자 전원이 자연스럽게 다 응답을 마치는 정상 케이스도 확인할 수 있도록 필수응답중/선택확인중 단계에서 참가자 행을 탭하면 그 사람의 응답완료 상태가 토글되는 데모 인터랙션을 추가함(실제 화면에는 없는 조작이며, 응답 자체는 참석자 각자의 화면(3·12)에서 이뤄짐 — 여기서는 조기 진행 결과만 시뮬레이션). 필수 참석자 전원이 응답완료되면 선택확인중으로, 선택 참석자 전원이 응답완료되면 압축 후보 중 "불가" 표시가 가장 적은 슬롯으로 자동 확정되어 스펙의 "전원 응답완료 시 마감 전이라도 조기 진행" 규칙을 그대로 재현함
  • PARTICIPANT.confirmation_status='확정'은 회의 시간이 최종 확정되는 순간 시스템이 전체 확정 참석자에게 자동으로 기록함(화면9의 "참석으로 변경"은 예외적 재전환 경로로 별도 유지)
