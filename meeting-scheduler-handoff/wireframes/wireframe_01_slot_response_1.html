구현 노트 (Claude Code 참고용)

  • 시간 슬롯 응답을 먼저 하고, 참석 형태(대면만/온라인만/둘 다 가능)는 그 다음(그리드 아래·footer 바로 위)에서 선택하도록 화면 순서를 변경함(회의당 1회, PARTICIPANT.attendance_mode). 이전 버전엔 참석 형태를 그리드 진입 전에 먼저 고르게 하고 미선택 시 그리드 위에 잠금 오버레이를 씌워 셀 클릭을 막았는데(pointer-events: none), "가능한 시간부터 먼저 표시하고 참석 형태는 나중에 정한다"는 흐름이 더 자연스럽다는 결정에 따라 순서를 바꾸고 그리드 잠금 기능 자체를 제거함 — 그리드는 화면 진입 즉시 편집 가능. "응답 완료" 버튼은 기존과 동일하게 참석 형태를 아직 고르지 않았으면 비활성화. 슬롯 응답·참석 형태 모두 마감 시각 전까지 몇 번이든 자유롭게 재수정 가능하며 최초 "응답 완료" 이후에도 잠기지 않음 — 단, 이미 한 번 "응답 완료"를 마친 뒤 참석 형태를 다시 바꾸면 "참석형태가 변경되어 회의 형태가 다시 계산돼요" 토스트가 떠서 재계산이 트리거된다는 것을 시각적으로 알려줌(실제로는 필수 참석자 전원 응답 완료 시점 혹은 마감 시점에 백엔드가 MEETING.mode를 재계산 — 이 화면엔 계산된 회의 형태를 직접 보여주는 지표가 없어 데모에서는 토스트만 시뮬레이션하고, 실제 갱신된 값은 화면 4·6에서 확인)
  • 확정 안내 배너는 "응답 완료" 버튼 바로 위(footer 영역)에 위치 — 원래 참석 형태 선택 아래 상단에 뒀었는데, 위쪽에 참석형태·안내·범례·개인정보 문구가 쌓이면서 grid-wrap(flex:1)이 눌려 .grid-scroll의 min-height:120px보다 작아지고, 그 결과 그리드가 스크롤힌트·footer 위로 겹쳐 보이는 실제 렌더링 버그가 있었음(844px 고정 높이 카드 안에서 위쪽 고정 콘텐츠 총합이 너무 커서 발생). 배너를 응답을 마무리하는 시점(버튼 직전)으로 옮기고 문구를 "확정 후엔 일정 변경이 어려워요 · 제출하신 일정은 시스템이 자동으로 조율해요" 한 줄로 압축했으며, 함께 .instruction·.legend·.privacy-note의 문구와 패딩도 줄여 각각 1~2줄 이내로 맞춤 — 결과적으로 그리드 표시 영역이 약 113px→207px로 늘어나 겹침 없이 6줄 이상 보임. 참고: 이전 초안에 있던 "AI가 확인 후 조율해요" 문구는 project_handoff.md의 "AI 활용 범위" 절(회의실 예약·회의 형태 계산·알림·리마인드는 전부 규칙 기반 자동화이며 LLM 불필요, LLM이 실제로 유용한 곳은 v2의 회의록 초안 생성·액션아이템 추출뿐)과 배치되어 이번에 규칙 기반 자동화를 뜻하는 "시스템이 자동으로 조율" 문구로 교체함 — 이 앱에서 LLM이 실제로 관여하는 화면은 없음
  • "응답 완료" 버튼: 참석 형태 선택 전엔 비활성화, 선택 후 활성화. 클릭 시 확인 팝업(선택한 참석 형태 요약 + "미표시 슬롯은 가능 처리" 안내)이 뜨고, "확인"을 눌러야 PARTICIPANT.responded_at이 기록됨(이 와이어프레임에선 로컬 변수로 시뮬레이션). 기록 후에도 그리드·참석 형태 모두 계속 편집 가능 — "완료"가 잠금을 의미하지 않음, 단지 "다 확인했다"는 명시적 신호. 완료 후엔 하단에 "✓ 응답 완료됨" 안내 문구가 추가로 표시됨
  • 필수 참석자가 마감 시각까지 참석 형태를 응답하지 않으면 시스템이 자동으로 "무관"으로 간주하고 회의 형태 계산을 진행함(진행이 막히지 않도록) — 한 명이라도 무응답으로 남아 계산이 영영 보류되는 일이 없게 하기 위한 규칙. 이 와이어프레임 자체는 마감 시각 경과를 실제로 시뮬레이션하는 타이머가 없으므로, 실제 구현 시 백엔드 배치 잡이 response_deadline 경과 시점에 PARTICIPANT.attendance_mode가 비어있는 필수 참석자를 either(무관)로 채운 뒤 회의 형태 계산을 트리거해야 함
  • 그리드 셀 = TIME_SLOT 1개. 30분 단위로 생성 (예: 13:30 시작 슬롯도 가능)
  • 후보 날짜는 항상 월요일 시작이 아님 — MEETING.created_at 다음 영업일부터 계산 (주말 제외, 이미 지난 날짜는 생성 안 함). 화·수요일 등 언제 생성해도 그 다음 영업일부터 후보가 시작됨
  • 요일 헤더에 날짜도 함께 표시 (예: "목 7/10") — 요일만으로는 어느 주인지 헷갈릴 수 있어서 병기
  • 셀 클릭 → SLOT_RESPONSE upsert (time_slot_id, user_id, status). 클릭마다 기피→불가→가능 순환(현재 상태가 "가능"인 셀을 클릭하면 기피로, 기피 상태에서 클릭하면 불가로, 불가 상태에서 클릭하면 다시 가능으로 돌아감)
  • SLOT_RESPONSE 레코드가 없는 셀 = 기본값 "가능" (별도 레코드 생성 안 함) — 단, 화면에는 옅은 체크(✓)로 명시해서 "응답 안 함"과 "가능으로 응답함"이 헷갈리지 않게 함. 안내 문구도 "모든 시간은 기본적으로 가능으로 표시돼 있다"는 전제를 먼저 알리고 "예외(기피·불가)만 클릭해서 표시"하라는 흐름으로 수정함 — 기존엔 "가능(✓)→기피(△)→불가(✕)" 3단계를 순서대로 거쳐야 하는 것처럼 읽혀서, 이미 전부 가능 처리된 화면에서 실제로는 첫 클릭에 기피가 뜨는 것과 문구가 어긋났음
  • 개인정보 안내 문구 추가: 범례 아래에 "내 응답 내용(기피·불가 표시 포함)은 다른 참석자에게 공개되지 않아요. 주최자에게도 집계된 결과만 전달돼요" — screen_specs.md 화면 5(대시보드)·6(완화요청 응답) 스펙 기준으로도 이미 참이었던 내용(대시보드엔 참가자별 responded_at 유무와 슬롯별 집계 점수만 노출, 완화요청 응답 화면은 본인이 설정한 제약만 본인에게 노출 — 어느 화면도 "누가 특정 슬롯에 기피/불가를 표시했는지"를 다른 사람에게 보여주지 않음)을 사용자에게도 명시적으로 알려주는 문구. 향후 화면 5·9 등을 만들 때도 이 원칙(개인별 응답 내용 비공개, 집계치만 공개)을 유지해야 함
  • 그리드 셀 자체는 제출 버튼 없이 클릭 즉시 저장(클릭마다 짧은 "저장됨" 토스트로 피드백, 0.9초 후 자동 사라짐). 하단에 이를 별도로 설명하는 고정 문구는 두지 않음 — 바로 아래 "응답 완료" 버튼과 나란히 있으면 "제출 버튼이 없다"는 문구와 실제 버튼이 동시에 보여 혼란을 줄 수 있어 제거함(그리드 자동저장과 "응답 완료"는 서로 다른 개념: 전자는 셀 값 저장, 후자는 responded_at 기록). response_deadline까지 미응답 시 리마인드 발송(별도 배치 작업)
  • 상단 우측은 USER.name, USER.department·USER.rank·USER.position, 그 아래 PARTICIPANT.role 배지 순으로 표시 — "내가 누구로서 응답하는지"를 명확히 하기 위함. 이름 앞 원형은 USER.profile_image_url 있으면 사진, 없으면 이름 첫 글자 이니셜 (앱 전체 공통 아바타 규칙)
  • "안건 등록" 탭은 별도 화면(화면 4)이며 같은 상위 진입점을 공유하는 구조
  • 요일 헤더는 스크롤 영역과 물리적으로 분리된 별도 테이블(고정, 스크롤 안 됨). 시간 그리드만 담긴 아래쪽 테이블만 세로 스크롤됨 — 두 테이블은 `<colgroup>`으로 열 너비를 동일하게 맞춰 정렬 유지
  • 모바일 기준 폭 390px (표준 아이폰 기준). 그리드 요일 열은 고정 px가 아니라 calc((100% - 시간열너비) / 5)로 계산해 화면 폭이 달라져도 항상 꽉 차게 반응
  • 아이콘은 Tabler Icons 라인 스타일로 통일 — 이모지·도형 placeholder 사용 안 함. 이전 버전은 CDN 아이콘 웹폰트(<i class="ti ti-...">)에 의존해 네트워크가 막힌 환경(오프라인 미리보기 등)에서 아이콘이 안 보이는 문제가 있었음 — 이번 수정에서 wireframe_03_meeting_create.html과 동일하게 인라인 SVG로 전부 교체해 외부 리소스 없이도 항상 렌더링되도록 함(상태바·뒤로가기·경고·잠금·스크롤 화살표·탭바 아이콘 전부 포함). 사람 아바타(이니셜/사진)는 별도 컴포넌트로 유지
  • 탭바 "알림" 아이콘의 빨간 점은 NOTIFICATION.is_read=false 항목이 있을 때만 표시(항상 떠 있는 게 아님). 이 와이어프레임에서는 데모로 항상 켜둠 — 실제로는 안건 등록/수정/삭제 시 작성자 본인 제외 나머지 참가자에게만 생성됨
