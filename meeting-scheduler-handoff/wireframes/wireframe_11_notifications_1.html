구현 노트 (Claude Code 참고용)

  • 접근 대상: 모든 로그인 사용자, 탭바 "알림" 탭. 이 화면은 top-level 탭 화면이라 다른 화면들과 달리 상단에 뒤로가기 화살표 없이 큰 제목(wireframe_05_meeting_list.html과 동일한 헤더 패턴)만 둠
  • 표시 순서·강조: NOTIFICATION 목록을 최신순으로 표시하고, is_read=false 항목은 스펙 문구 "굵게 또는 좌측 점" 중 둘 다 적용(좌측 빨간 점 마커 + 회의 제목/문구 굵게)해서 시인성을 더 확실히 함. 항목마다 관련 회의 제목·알림 문구·발생 시각을 표시하고, 어떤 종류의 변경인지 한눈에 알 수 있도록 NOTIFICATION.type 값을 작은 태그로도 함께 보여줌(스펙에 명시된 항목은 아니지만 10가지 타입이 뒤섞여 나열되므로 구분에 도움이 될 것 같아 추가함)
  • 항목 탭 시 동작: 클릭하면 (1) 토스트로 "관련 회의 화면으로 이동"을 시뮬레이션하고 (2) 해당 항목의 is_read를 true로 바꿔 좌측 점·굵게 표시가 즉시 사라짐 — 실제로는 페이지 이동이 일어나지만 이 와이어프레임은 알림 목록 자체가 초점이라 라우팅은 토스트로만 표현
  • "대시보드 또는 상세" 라우팅 규칙의 실제 적용 한계(스펙 보완 필요 지점): 스펙 원문은 이동 대상을 "대시보드 또는 상세" 두 가지로만 서술하지만, 실제로는 알림 종류·회의 진행 단계·보는 사람의 역할(주최자/참석자)에 따라 더 세분화된 화면으로 보내는 게 더 자연스러워 보임. 그래서 이 와이어프레임에서는 route 값을 "dashboard"/"detail" 두 개로 제한하지 않고, 안건수정 알림은 "agenda"(→ 화면 4), 완화요청(변경요청) 알림은 "response"(→ 화면 6, 완화요청 응답)로 세분화해 실제로 더 맞는 라우팅이 어떤 모습일지 시범적으로 보여줌. 대시보드(화면 5)는 주최자 전용이라 참석자 입장에선 애초에 접근 권한이 없다는 점도 스펙과 충돌하는 부분 — 실제 구현 전에 이 라우팅 규칙을 알림 타입별·역할별로 정식으로 다시 정의하는 걸 권장함
  • 탭바 배지 동기화: db_model_mvp.md·screen_specs.md 규칙대로 개수가 아닌 존재 여부만 표시하도록, 이 화면 자체의 탭바 "알림" 아이콘 빨간 점도 목록의 미확인 항목 수를 실시간으로 반영해 전부 읽으면 사라지게 만듦(다른 화면들의 탭바 점은 정적 표시이며, 이 화면만 유일하게 동적으로 갱신됨 — 알림 목록 화면이라는 맥락상 가장 자연스러운 곳이라 판단)
  • 상단 부제목은 "미확인 N건"으로 표시하다가 전부 읽으면 "모두 확인했어요"로 바뀜 — 스펙엔 없는 세부 표현이지만 사용자가 진행 상황을 바로 알 수 있도록 추가함
  • 데모 데이터는 NOTIFICATION.type 10종(안건등록/안건수정/시간확정/불참안내/재조율/참석재확인/회의실확정/완화요청/마감임박/응답요청 등) 중 다양한 케이스를 섞어 구성. "불참안내" 항목은 wireframe_09_meeting_detail.html의 "정다은님이 압축후보 전부 불가로 불참 처리된" 시나리오와 동일한 맥락을 재사용함. "재조율" 타입은 두 단계를 각각 보여줌 — ① 재조율이 막 시작된 시점("필수 참석자 불참으로 재조율이 시작됐어요")과 ② 새 시간이 정해진 시점("회의 시간이 O→O로 변경됐어요")을 서로 다른 알림으로 분리해 시간순으로 배치. "완화요청" 타입도 양쪽 당사자 시점을 모두 넣음 — 주최자가 보낸 직후 시점("이지훈님에게 요청했어요")과, 실제로 요청을 받은 당사자 시점("참석 가능 여부를 다시 확인해주세요", 화면 표시상 wireframe_07처럼 "변경요청"으로 순화된 문구)을 함께 예시로 둠
