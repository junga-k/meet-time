  • 이 화면은 원래 스펙상 v2로 연기됐던 "10. 회의록·액션아이템" 화면의 최초 제작본이며, 이번 라운드에서 사용자 피드백을 반영해 대폭 재설계했다. 상세 스펙 문서 v2_expansion_scope.md·데이터 모델 문서 db_model_full.md가 없어(db_model_mvp.md에도 MEETING_NOTE/ACTION_ITEM 미정의) 계속 자체 설계로 진행 중이며, 아래는 전부 현재까지의 가정/결정이다.
  • 화면 표시 용어 변경 — "액션아이템" → "향후 추진 과제"로 화면에 노출되는 텍스트만 교체(내부 데이터 모델 명칭은 계속 ACTION_ITEM 유지). 회의 후속 조치를 담당자·마감일과 함께 추적하는 투두(할 일) 목록이라는 의미는 동일.
  • 회의록 작성 가능 시점 갱신 — "회의 시작부터 실시간 작성 가능"으로 확정(기존엔 "회의 종료 후"만 가정). wireframe_09_meeting_detail_3.html의 진입 버튼도 회의가 시작된 순간(진행 중·종료 공통, meetingStarted())부터 "회의록 확인하기"로 활성화되도록 갱신함(데모 컨트롤도 참석 상태 × 회의 진행 상태 2개 축으로 분리해 모든 조합 확인 가능해짐).
  • 작성자 제도 신설: 회의록 작성자는 기본값 주최자(이 데모에서는 "김민준")이며, 주최자가 참석자 중 다른 사람을 작성자로 재지정할 수 있음("변경" 버튼 — 로그인 사용자가 주최자일 때만 노출). 편집 권한은 현재 지정된 작성자 1인에게만 있음 — 작성자가 아닌 참석자는 "회의 내용" textarea가 읽기 전용(회색 배경)으로 바뀌고, "이 회의록은 {작성자}님만 작성할 수 있어요" 안내 배너가 뜨며 AI 초안 생성·임시저장·등록 버튼도 전부 숨겨짐(열람은 항상 가능). 데모에서는 상단 [데모 전용] 컨트롤로 로그인 사용자를 김민준(주최자)/박서연/이지훈 중 바꿔가며 권한 차이를 확인할 수 있음. 향후 추진 과제 섹션은 이 제한과 무관하게 참석자 누구나 추가·완료체크·삭제 가능(회의록 작성 권한 제한은 회의록에만 적용).
  • 회의록 양식 필드 확정: 회의록 상단에 날짜·시간·참석인원(직위순 나열, 동일 직위는 이름 가나다순)·안건 제목 4개 필드를 MEETING/PARTICIPANT/AGENDA 데이터로 자동 채워 읽기 전용으로 표시(작성자도 이 필드는 직접 수정하지 않음). 실제로 작성자가 작성하는 부분은 그 아래 "회의 내용" 텍스트 영역 하나뿐.
  • 참석인원 정렬 규칙(재확인): "직위순"이 배열 순서가 아니라 실제 직위 서열 기준 정렬이어야 함을 확인해 RANK_ORDER(사장·부사장·전무·상무·이사·부장·차장·과장·대리·주임·사원 — 회사마다 체계가 다를 수 있어 이 데모 데이터에 등장하는 직위만 임시 순서표로 사용) 기준 정렬 함수(sortByRankThenName)를 추가해 렌더링. 동일 직위는 이름 가나다순(localeCompare(name, "ko"))으로 2차 정렬. 예: 과장 김민준 → 대리 박서연·최유나(가나다순) → 사원 이지훈·정다은·한소희(가나다순).
  • 안건 첨부파일 반영: 화면4(안건 등록, wireframe_02_agenda_1.html)에 이미 AGENDA.attachment_url 첨부파일 기능이 있으므로, 화면10의 "안건" 필드도 각 안건에 첨부파일이 있으면 파일명 칩(클립 아이콘+파일명, 클릭 시 데모 토스트로 다운로드 시뮬레이션)을 함께 표시하도록 보완. 첨부파일이 없는 안건은 제목만 표시.
  • "AI 초안 생성" 버튼의 정확한 동작 방식(사용자 질문에 대한 답): 녹음·음성인식(STT) 기반이 아니며, 작성자가 이미 "회의 내용" 칸에 러프하게 입력해둔 메모(문장이 안 갖춰진 키워드·조각글이어도 무방)를 원본으로 삼아, 실제 구현 시 LLM(예: Claude API) 호출을 통해 다듬어진 문단(요약·결정사항 등)으로 재정리해 같은 칸에 덮어써 채워 넣는 용도다. 위의 날짜·시간·참석인원·안건 같은 자동 채움 필드는 AI가 건드리지 않고 시스템이 그대로 유지한다. "회의 내용"이 비어 있으면 정리할 원본이 없으므로 버튼이 비활성화된다(이 와이어프레임 데모에서는 실제 LLM 호출 없이 700ms 지연 후 고정 예시 문구로 대체 — 이미 내용이 있으면 "기존 내용을 덮어쓸까요?" 확인 팝업을 먼저 띄움).
  • 저장 흐름을 "임시저장" / "등록" 2단계로 분리: 등록 전까지는 하단에 "임시저장"(보조)·"등록"(주요) 버튼이 함께 노출되며, 임시저장은 몇 번이든 자유롭게 반복 가능하고(참석자 전원 알림 없음, 작성자 본인 및 다른 참석자 모두 그 시점까지의 내용을 언제든 다시 열어 확인 가능), "등록"을 누르는 순간에만(확인 팝업 통과 후) 회의록이 최종 등록되고 해당 회의 참석 인원 전체에게 "회의록 등록" 안내 알림(NOTIFICATION)이 발송된다. 등록 이후에는 버튼이 "저장" 하나로 바뀌어 계속 내용을 수정할 수 있지만, 수정 시마다 다시 알림을 보내지는 않는 것으로 가정(스팸 방지 목적 — 재검토 여지 있음). notesMeta 영역에 "임시저장됨 · {시각}" 또는 "등록됨 · {시각} · 참석자 전원에게 알림 발송됨"을 표시.
  • 제안 데이터 모델(신규, 문서 미반영): MEETING_NOTE(meeting_id, content, author_user_id, is_ai_generated, status: 임시저장/등록, registered_at, updated_at).
